# RWR Editor Next 兼容性契约与回归基线

本文档面向后续维护者，规定 RWR Editor Next 在修改解析器、编辑逻辑、动画系统或桌面文件接口时不得破坏的行为。它不是界面设计说明，也不要求复刻 2011 年原版 RWR Editor 的外观和缺点；它关心的是：原版能够处理的 RWR 资源，在 Next 中编辑后仍能被原版编辑器及游戏正确读取。

## 1. 判定依据

发生冲突时，按以下优先级判断兼容性：

1. **游戏实际读取结果**：目标版本的 _Running With Rifles_ 能否正确载入并显示资源，是最高标准。
2. **2011 年原版 EXE 的可观察行为**：同一输入、同一编辑操作产生的模型含义，是历史基线。
3. **已知可用的社区资源与官方资源**：真实模型和动画比人工构造的“理想格式”更有代表性。
4. **本仓库的自动化测试和固定样例**：用于阻止已经确认的行为回退。
5. **当前实现细节**：仅在以上依据均未覆盖时作为参考，不能反过来定义文件格式。

“原版能打开”不等于“必须逐字节输出相同 XML”。兼容性默认按解析后的结构、数值和游戏表现判断；空白、缩进、属性顺序及小数文本形式可以不同。

本文中的关键词含义如下：

- **必须**：不满足即视为兼容性缺陷，不应发布。
- **应当**：通常需要满足；若暂时不能满足，应在提交或发行说明中记录原因和影响。
- **可以**：允许实现，但不能破坏“必须”项。

## 2. 原版 EXE 提供的功能基线

原版 RWR Editor 至少确立了以下核心工作流，Next 必须保留其结果语义：

| 工作流     | 兼容要求                                                           |
| ---------- | ------------------------------------------------------------------ |
| 打开模型   | 能读取原版可用的模型 XML，并显示体素颜色、位置和已有骨骼结构。     |
| 体素编辑   | 能选择、添加、删除、绘色、取色及沿坐标轴移动体素。                 |
| 保存模型   | 保存结果仍是可被原版和游戏读取的模型 XML。                         |
| 骨骼与绑定 | 能读取骨骼点、骨骼连线及体素绑定，不因普通体素编辑产生悬空索引。   |
| 动画载入   | 能读取原版动画 XML 中的动画、关键帧和骨骼点位置。                  |
| 视图检查   | 能旋转、平移和缩放视图，观察模型及骨骼。视图操作不得修改模型数据。 |

原版的启动配置窗口、阻塞式文件选择、固定界面布局、低可读性配色、旧式快捷键限制和只能观察骨骼帧等问题，**不是**需要保留的兼容行为。Next 可以改进交互、渲染和预览，只要不改变保存后的资源含义。

## 3. 模型 XML 契约

### 3.1 最低可接受结构

当前解析器接受以下基本结构：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<model>
  <voxels>
    <voxel x="0" y="0" z="0" r="1" g="0" b="0" a="1"/>
  </voxels>
  <skeleton>
    <particle id="0" name="a" x="0" y="0" z="0" invMass="1" bodyAreaHint="1"/>
    <particle id="1" name="b" x="1" y="0" z="0" invMass="1" bodyAreaHint="1"/>
    <stick a="0" b="1"/>
  </skeleton>
  <skeletonVoxelBindings>
    <group constraintIndex="0"><voxel index="0"/></group>
  </skeletonVoxelBindings>
</model>
```

模型文件必须满足：

- 根元素是 `<model>`，并包含直接子元素 `<voxels>`。
- `<voxel>` 的 `x/y/z` 是模型坐标；`r/g/b/a` 是浮点颜色分量。常规资源使用 `0` 到 `1`，但读取阶段不得擅自把合法有限数值量化成 8 位颜色。
- 缺失的 `x/y/z` 当前按 `0` 读取；缺失的 `r/g/b/a` 当前按 `1` 读取。改变该回退规则前必须增加真实资源依据和迁移测试。
- `a="0"` 必须保持完全透明，不能因真假值转换而变成 `1`。
- `<skeleton>` 和 `<skeletonVoxelBindings>` 可以为空或缺失；没有骨骼的普通体素模型仍然有效。
- `<particle>` 的 `id` 用于解析 `<stick a="…" b="…"/>` 的端点，不能把数组下标误当成粒子 ID。
- `<group constraintIndex="n">` 中的 `constraintIndex` 指向 `<skeleton>` 内按文档顺序排列的第 `n` 个 `<stick>`。
- `<group>` 中的 `<voxel index="n"/>` 指向 `<voxels>` 内按文档顺序排列的第 `n` 个体素。

### 3.2 读取与保存

- UTF-8 和带 UTF-8 BOM 的 XML 必须可读；桌面边界在交给解析器前移除 BOM。
- 保存必须输出 UTF-8 XML 声明。
- 模型保存会重新生成 `<voxels>` 和 `<skeletonVoxelBindings>`，因此这两个区域内只保证标准字段的语义；不要依赖其中的原始缩进、属性顺序或非标准属性。
- 模型中这两个区域以外的元素、属性、注释及骨骼定义应当在一次“打开—保存”后保持语义不变。
- 体素坐标按整数文本或最多六位小数输出；颜色分量固定输出六位小数。这是文本规范化，不应改变其数值含义。
- 保存时必须根据当前体素顺序重新计算绑定索引。禁止直接沿用载入时的旧数字索引。
- 仅格式化差异不算不兼容；节点丢失、数值变化、绑定对象变化或游戏表现变化才算不兼容。

### 3.3 编辑不变量

- 每个已载入体素在编辑会话中必须拥有稳定的内部身份。移动、绘色和撤销不得通过易变化的数组下标识别体素。
- 新增体素不得占用已有体素的完全相同坐标。
- 批量移动是原子操作：只要任一目标位置被未选体素占用，整次移动都不执行。
- 删除体素时必须同步清理所有指向它的绑定；保存后不得出现超出体素数量的引用。
- 绘色只修改 `r/g/b`；除非用户明确执行透明度相关操作，否则不得意外修改原有 `a`。
- 撤销和重做必须同时恢复体素内容及绑定关系。
- 新建模型的用户界面默认创建位于 `(0, 0, 0)` 的一个基点体素。核心层仍支持创建八体素基块，但它不是当前默认工作流。
- “重新绑定到最近骨骼”是 Next 的辅助功能：它按体素点到每条骨骼线段的最短距离分组。该操作必须可撤销，且不能在用户未明确触发时自动执行。

## 4. 动画 XML 契约

### 4.1 已支持结构

动画文件以 `<animations>` 为根，每个直接子元素 `<animation>` 使用：

- `comment`：显示名称；
- `loop`：`"0"` 表示不循环，其它值或缺失值当前表示循环；
- `end`：动画结束时间；
- `speed`：播放速度，缺失时为 `1`；
- `speed_spread`：速度随机范围，缺失时为 `0`；
- `<frame time="…">`：关键帧；
- `<position x="…" y="…" z="…"/>`：按模型 `<particle>` 顺序排列的骨骼点位置。

动画导出必须：

- 保持上述字段的数值和含义；
- 按 `time` 从小到大输出关键帧；
- 以六位小数输出动画及坐标数值；
- 保持每帧位置顺序与模型骨骼点顺序一致；
- 能被本项目重新解析，并能与对应模型共同预览。

当前动画序列化器会根据内存中的已知字段重建整个 `<animations>` 文档，**尚不保证**保留未知动画属性、未知子节点和注释。若社区资源依赖这些扩展，维护者应先添加固定样例，再扩展解析和保存逻辑，不能静默丢弃后直接宣称兼容。

### 4.2 预览与编辑不变量

- 关键帧之间使用线性插值；时间早于首帧时使用首帧，晚于末帧时使用末帧。
- 循环动画到达 `end` 后回绕；非循环动画停在结束姿势。
- 缺少某个骨骼点的位置时，预览应使用该点已有位置或模型绑定姿势，不能产生 `NaN`、无限值或使整个模型消失。
- 动画预览是只读计算，不得修改模型中保存的体素坐标。
- `constraintIndex` 决定体素跟随哪条 `<stick>`。预览时，体素沿骨骼轴的位置比例及相对骨骼轴的径向偏移应当随骨骼旋转和平移。
- 对长度接近零的骨骼线段，预览退化为随首端点平移，不能生成无效旋转。
- 只有“创建与编辑”页面允许拖动骨骼点并修改当前关键帧；普通预览和人物效果预览不得写回模型或动画。
- 动画修改与模型修改使用独立的未保存状态。保存模型不能暗中覆盖动画文件，导出动画也不能暗中保存模型。

人物模型效果预览、内置典型动作、固定摄像机、草地和光照是 Next 的展示能力，不属于原版文件格式。它们可以迭代，但必须复用同一套模型绑定与动画采样语义，不能发展出第二套不一致的解析器。

## 5. 文件与桌面边界

- 模型和动画通过异步原生文件对话框选择；等待用户选择时主窗口不得进入“未响应”。
- 用户取消打开或保存时返回空结果，当前模型、动画、路径和未保存状态都不得改变。
- 另存为未填写 `.xml` 扩展名时自动补全。
- 覆盖保存只允许写入本次应用会话中由用户打开或成功另存过的现有 XML 文件。
- 覆盖路径必须经过规范化并确认为普通 `.xml` 文件；前端传入任意路径不能绕过该限制。
- 浏览器开发模式可以使用文件输入和下载作为回退，但核心解析与序列化结果必须与 Tauri 版本一致。
- 文件对话框、磁盘读写、安装包和系统窗口属于 Tauri/Rust 边界；模型及动画语义应保持在与平台无关的 TypeScript 核心层。

## 6. 允许变化与禁止变化

### 可以自由改进

- Vue 组件结构、CSS、主题、字体和响应式布局；
- 工具位置、提示文字、快捷键及可配置项；
- Three.js 材质、辅助网格、灯光和非数据性的显示模式；
- 摄像机控制、框选、确认提示和人物效果预览；
- 文件对话框在不同操作系统上的外观。

### 必须先提供兼容证据

- 改变坐标轴含义、单位、原点或体素尺寸；
- 改变颜色分量范围、透明度默认值或数值精度；
- 改变粒子、骨骼线段、绑定组或动画位置的索引规则；
- 丢弃、重排或自动修正未知 XML 内容；
- 自动合并重复体素、自动重绑定或自动改变骨骼；
- 让预览计算写回原始模型数据；
- 为方便渲染而改变最终保存坐标。

若确实需要进行以上变更，提交中必须包含真实输入样例、原版或游戏中的观察结果、修改前后的结构对比和自动化回归测试。

## 7. 建立原版黑盒样例的方法

当遇到源码和现有测试无法解释的旧资源时，不要凭直觉“清理格式”。应按以下流程建立证据：

1. 复制问题 XML，删去与问题无关的内容，制作最小复现样例；保留未经修改的原件。
2. 在原版 EXE 中打开样例，记录能否载入、显示结果和错误信息。
3. 分别执行一次最小操作，例如移动一个体素、删除一个体素、修改一种颜色或保存但不编辑。
4. 使用“另存为”保存，比较解析后的 XML 树和关键数值，不只比较文本行。
5. 将原版输出放入游戏验证；若原版与游戏冲突，以游戏读取结果为准并记录游戏版本。
6. 把脱敏且可再分发的最小样例加入 `tests/fixtures/`，再写一个只验证该行为的测试。
7. 在测试名称或注释中说明样例来源、原版操作和预期结果，避免后来者误删“看似奇怪”的兼容逻辑。

除非普通输入/输出观察无法回答问题，否则不需要反编译原版 EXE。即使进行了逆向，也只应把观察到的格式规则写成独立测试，不复制来源不明或许可不兼容的代码。

## 8. 回归测试最低矩阵

### 自动化检查

每次修改解析、保存、绑定或动画代码，至少执行：

```sh
bun run test
bun run build:frontend
bun run test:rust
```

发布构建执行：

```sh
bun run build
```

现有测试至少覆盖：

- 透明体素读取；
- 移动后的稳定绑定；
- 删除后的绑定清理；
- 最近骨骼重新绑定；
- 模型非体素内容的语义保留；
- 新模型生成；
- 动画元数据、关键帧坐标和往返序列化；
- 绑定体素随骨骼移动和旋转；
- 动画关键帧线性采样；
- Tauri 桌面命令映射、Vue DOM 契约及关键交互算法。

### 发布前人工检查

至少选择三类真实资源：无骨骼模型、已绑定人物模型、包含多个动画的动画文件。对每类执行适用项目：

1. 打开后核对体素数量、颜色、骨骼点、骨骼线段和绑定数量。
2. 不编辑直接另存，重新打开并核对结构与显示。
3. 分别测试添加、删除、绘色、移动、撤销、重做和批量选择。
4. 覆盖保存前取消一次，确认文件和未保存状态未改变；再确认保存并重新打开。
5. 载入动画，检查首帧、中间插值、末帧、循环和非循环行为。
6. 检查未绑定体素保持原位，已绑定体素随正确骨骼运动。
7. 在支持的目标系统上验证非阻塞文件对话框、便携程序和安装包。
8. 对发布候选产物至少进行一次游戏内载入检查。

## 9. 当前已知边界

- 自动化测试验证的是已收集样例和明确规则，不代表覆盖所有历史社区 XML 变体。
- 模型 XML 只对 `<voxels>` 和 `<skeletonVoxelBindings>` 的标准内容做往返保证；其中的非标准扩展尚无保留承诺。
- 动画 XML 当前只保留编辑器已识别的字段，未知扩展内容可能在导出时丢失。
- 人物效果预览用于快速检查，不是游戏渲染器的逐像素复刻；最终兼容性仍需游戏内验证。
- 跨平台 WebView 允许存在字体栅格、文件对话框和少量布局差异，但不允许因此改变 XML、坐标、颜色、绑定或动画数值。

如果新发现的原版行为与本契约冲突，应先保存可复现证据并更新测试，再修改本文档。不要只为了让当前实现“看起来正确”而降低兼容要求。
